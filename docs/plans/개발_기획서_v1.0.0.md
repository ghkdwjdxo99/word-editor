현재 시제품을 첫 정식 배포판인 v1.0.0으로 안정화하는 기획이다. 새 편집 기능을 늘리기보다 데이터 유실 방지, DOCX 호환성 검증, DOC 시험 검증 및 독립 실행 배포 준비를 우선한다.

- 새 문서 변경 표시 정책 확정

- 오류 메시지 분류 개선
- RichTextBox 변환 자동 회귀 테스트 보강
- Microsoft Word 및 LibreOffice DOCX 호환성 검증
- Word 설치 환경의 DOC 왕복 검증
- Windows x64 자체 포함 배포본 검증

### v1.0.0에서 제외할 변경

- 표와 이미지의 원형 편집

- 맞춤법 검사
- 클라우드 및 공동 편집

## 1. 검토 결론

현재 프로그램은 “직접 작성한 단순 문서를 DOCX로 저장하고 다시 여는 기능”까지는 구현되어 있다. 빌드와 현재 자동 검증 12건도 모두 통과한다.

그러나 아직 일반 사용자에게 배포할 단계는 아니다. 화면에 표시된 텍스트가 저장 파일에서 사라질 수 있는 경로가 있고, Word/LibreOffice 실제 호환성 및 DOC 연동이 검증되지 않았다. 다음 개발의 최우선 목표는 기능 추가가 아니라 **사용자가 보는 본문을 절대 조용히 잃지 않도록 만드는 것**이다.

현재 판정은 다음과 같다.

- 내부 시연: 가능
- 제한된 개발 테스트: 가능
- 일반 사용자 배포: 보류
- 배포 전 필수 조건: P0 데이터 유실 위험 제거 및 실제 오피스 호환성 확인

## 2. 검토 범위와 확인 결과

다음 항목을 소스와 기존 기획서 기준으로 검토했다.

- WPF 화면과 편집 명령
- 문서 상태 및 미저장 확인
- RichTextBox와 내부 문서 모델 간 변환
- DOCX ZIP/XML 읽기·쓰기
- 임시 파일을 사용한 안전 저장

- 기존 테스트 결과 보고서

재검증 결과는 다음과 같다.

## 3. 현재 구현된 기능

### 3.1 충분히 구현된 부분

- 새 문서, 열기, 저장, 다른 이름으로 저장
- DOCX 최소 패키지 생성 및 기본 검증
- 문단, 빈 문단, 한글/영문/숫자, 탭, 줄바꿈 처리
- 굵게, 기울임, 밑줄, 8~72pt 글자 크기
- 왼쪽, 가운데, 오른쪽 문단 정렬
- 실행 취소, 다시 실행, 잘라내기, 복사, 붙여넣기
- 변경된 문서를 닫거나 교체하기 전 저장 여부 확인
- 지원하지 않는 DOCX 요소의 일부 감지 및 경고
- 임시 파일 검증 후 원본 교체
- Word가 설치된 경우 DOC↔DOCX 변환을 시도하는 구조
- 오류 로그에 문서 본문을 기록하지 않는 기본 정책

### 3.2 구현은 되었지만 검증이 부족한 부분

- Word COM을 통한 DOC 열기와 저장
- Word 프로세스와 COM 객체의 완전한 정리
- Microsoft Word에서 생성 DOCX 열기
- LibreOffice Writer에서 생성 DOCX 열기
- 저장/폐기/취소 대화상자의 실제 사용자 흐름
- 파일 잠금, 읽기 전용, 디스크 공간 부족 상황
- 다양한 외부 프로그램에서 복사한 내용 붙여넣기

## 4. 부족한 부분과 위험도

## P0 — 배포 전에 반드시 해결

### P0-1. 중첩 서식 텍스트가 저장 시 사라질 수 있음

현재 편집기→문서 모델 변환은 문단 바로 아래에 있는 `Run`과 `LineBreak`만 처리한다. `Span`, `Bold`, `Italic`, `Underline`, `Hyperlink` 같은 컨테이너 안의 텍스트는 화면에 보이더라도 저장 대상에서 빠질 수 있다.

주요 발생 경로는 다른 프로그램이나 웹 페이지에서 서식 있는 텍스트를 붙여넣는 경우다. 이는 사용자 본문 유실이므로 가장 먼저 고쳐야 한다.

요구사항:

- 모든 인라인 컨테이너를 재귀적으로 순회한다.
- 부모의 굵게/기울임/밑줄/글자 크기를 자식 텍스트에 상속한다.
- 지원하지 않는 링크 기능은 제거하더라도 링크의 표시 텍스트는 보존한다.
- 변환할 수 없는 인라인이 있어도 화면의 일반 텍스트는 최대한 보존한다.
- 저장 전후 본문 문자열이 달라지면 저장을 중단하거나 명확히 경고한다.

완료 기준:

- 중첩 Span 3단계, Bold/Italic/Underline 조합, Hyperlink 표시 문자열이 왕복 후 유지된다.
- Word, 브라우저, 메모장 등에서 붙여넣은 본문이 저장 후 누락되지 않는다.

### P0-2. 표·목록 등 블록 콘텐츠가 조용히 누락될 수 있음

현재 변환기는 편집 화면의 최상위 `Paragraph`만 저장한다. 붙여넣기로 WPF `Table`, `List`, `Section` 등이 들어오면 화면에는 보일 수 있지만 저장할 때 해당 블록 전체가 빠질 수 있다.

더 큰 문제는 기존 DOCX를 열 때만 비지원 콘텐츠 상태가 설정되고, 사용자가 편집 중 새로 붙여넣은 비지원 블록은 경고 상태에 반영되지 않는다는 점이다.

MVP 권장 정책:

- 편집기에 붙여넣을 때 서식 있는 콘텐츠를 일반 텍스트와 지원 서식만 남도록 정규화한다.
- 표는 셀 텍스트를 탭과 줄바꿈으로 평탄화하거나 붙여넣기 전에 “표 서식은 제거되고 텍스트만 붙여넣습니다”라고 안내한다.
- 목록은 각 항목의 본문을 일반 문단으로 변환한다.
- 편집 문서 전체를 저장 직전에 검사하여 비지원 블록이 있으면 저장을 중단하고 안내한다.
- 아무 경고 없이 비지원 블록을 버리는 동작은 금지한다.

완료 기준:

- 표, 목록, Section이 포함된 붙여넣기 시 본문 텍스트가 보존되거나 저장 전에 차단된다.
- 화면에 있던 본문이 경고 없이 저장 결과에서 사라지는 테스트가 0건이다.

### P0-3. 실제 DOCX 호환성을 확인하지 않음

내부 판독기로 다시 여는 테스트만으로는 DOCX 호환성을 증명할 수 없다. 생성 XML이 프로그램 자신의 판독기에는 맞아도 Microsoft Word나 LibreOffice에서 복구 경고가 나타날 수 있다.

완료 기준:

- Microsoft Word 최신 데스크톱 버전에서 복구 경고 없이 열린다.
- LibreOffice Writer에서 복구 경고 없이 열린다.
- 한글, 공백, 빈 문단, 탭, 줄바꿈, 지원 서식과 정렬이 두 프로그램에서 육안 확인된다.
- 두 프로그램에서 다시 저장한 DOCX를 편집기가 열 수 있다.
- 검증 프로그램과 버전, 결과 화면을 테스트 보고서에 남긴다.

### P0-4. DOC 기능이 구현만 되고 실제 검증되지 않음

DOC는 Word 설치 환경에 의존한다. 현재 코드가 존재한다는 사실만으로 지원 기능이라고 표시하면 안 된다.

검증 전 제품 정책:

- DOC 기능을 “시험적 기능”으로 표시하거나 배포 빌드에서 숨긴다.
- README의 지원 문구에도 Word 버전 및 미검증 상태를 표시한다.

완료 기준:

- Word 설치 PC에서 단순 DOC 열기→수정→DOC 저장→Word 재열기가 성공한다.
- 저장 후 원본 형식이 실제 Word 97-2003 DOC인지 확인한다.
- 성공, 변환 실패, 암호 문서, 읽기 전용 문서를 각각 시험한다.
- 작업 후 새 `WINWORD.EXE` 프로세스와 임시 DOCX가 남지 않는다.
- 사용자가 이미 열어 둔 Word 문서나 Word 프로세스에는 영향을 주지 않는다.

## P1 — 첫 배포 품질을 위해 해결

### P1-1. 새 문서 상태 표시가 기획과 다름

기획서는 저장 전 새 문서 제목에 `*`를 표시하도록 했지만 초기화 직후 변경 상태는 false다. XAML 초기 제목의 별표도 생성자 실행 중 다시 제거된다.

결정이 필요하다.

- 권장안: `*`는 “마지막 저장 이후 변경됨”만 의미하게 하고, 새 빈 문서는 별표 없이 표시한다.
- 대안: “아직 저장된 적 없음”까지 별표 의미에 포함해 초기 새 문서에도 표시한다.

권장안을 택하면 코드를 바꾸기보다 기획서와 테스트 기대값을 수정한다. 사용자가 실제 입력하면 지금처럼 별표를 표시한다.

### P1-2. 저장 확장자 검증이 약함

저장 대화상자에서 사용자가 지원하지 않는 확장자를 직접 입력했을 때, 현재 로직은 DOC가 아닌 모든 경로를 DOCX로 처리할 가능성이 있다. 그 결과 내용은 DOCX인데 확장자는 `.txt` 같은 잘못된 파일이 생길 수 있다.

요구사항:

- 저장 대상은 `.docx` 또는 Word 연동 가능 시 `.doc`만 허용한다.
- 확장자가 없으면 선택한 필터에 맞게 추가한다.
- 다른 확장자를 입력하면 자동 변경 전에 사용자 확인을 받는다.
- 파일 내용과 확장자가 다른 결과물을 만들지 않는다.

### P1-3. 선택 상태 도구 모음이 불완전함

선택 변경 시 굵게와 기울임 버튼만 갱신한다. 밑줄, 글자 크기, 문단 정렬은 현재 커서 위치나 선택 내용과 UI 표시가 맞지 않을 수 있다.

요구사항:

- 커서 이동과 선택 변경 시 모든 서식 컨트롤을 동기화한다.
- 서로 다른 서식이 섞인 선택은 “혼합 상태”로 표시한다.
- 버튼 클릭 때문에 선택 영역이 사라지거나 예기치 않은 서식이 적용되지 않아야 한다.

### P1-4. 오류 분류가 실제 예외 구조와 맞지 않을 수 있음

DOCX 판독기는 여러 파일/권한 예외를 `InvalidDataException`으로 감싼다. 화면 오류 처리기는 바깥 예외 형식만 보고 손상 문서라고 안내하므로, 권한 문제나 파일 잠금을 손상으로 오인할 수 있다.

요구사항:

- 사용자 행동 기준으로 오류 범주를 정의한다: 권한, 점유, 손상, 공간 부족, 미지원, Word 연동 실패.
- 내부 예외까지 확인해 가장 정확한 안내를 표시한다.
- 상세 예외는 로그에, 쉬운 해결 방법은 대화상자에 표시한다.

### P1-5. 테스트가 서비스 계층에 편중됨

현재 테스트 프로젝트는 별도 테스트 프레임워크 없이 콘솔에서 조건을 확인한다. 다운로드를 피한다는 목표에는 맞지만, RichTextBox 변환과 MainWindow 사용자 흐름은 거의 자동 검증하지 않는다.

외부 패키지 없이 개선하는 방안:

- 현재 콘솔 검증 방식을 유지한다.
- STA 스레드에서 RichTextBox를 생성하는 변환 테스트를 추가한다.
- 저장/열기 대화상자 로직을 UI에서 분리해 순수 상태 전이로 검증한다.
- 테스트마다 고유한 임시 폴더를 사용하고 종료 시 정리한다.
- 실패 시 테스트 이름, 예상값, 실제값을 출력한다.

## P2 — 안정화 이후 개선

### P2-1. 일반적인 Word 문서 표현의 한계

- 스타일 기반 글자 크기와 서식을 해석하지 않는다.
- 글꼴, 색상, 강조, 위첨자/아래첨자를 보존하지 않는다.
- 양쪽 정렬을 왼쪽 정렬로 처리한다.
- 하이퍼링크 주소는 보존하지 않는다.
- 목록, 표, 이미지의 원형 보존은 지원하지 않는다.

이 항목들은 MVP 범위 밖이므로 즉시 구현하기보다 열기 화면과 저장 경고에서 손실 범위를 명확히 알리는 것이 우선이다.

### P2-2. 배포 방식이 확정되지 않음

현재 대상 프레임워크는 `net10.0-windows`다. 대상 PC에 해당 .NET Desktop Runtime이 없다면 실행되지 않는다. “웬만하면 다운로드하지 않음”이라는 목표는 개발 패키지뿐 아니라 사용자 실행 환경에도 적용해야 한다.

결정안:

| 방식 | 장점 | 단점 |
|---|---|---|
| 프레임워크 종속 배포 | 용량이 작음 | 대상 PC에 정확한 .NET Desktop Runtime 필요 |
| 자체 포함 배포 | 별도 런타임 설치 불필요 | 배포 용량이 큼 |
| 단일 파일 자체 포함 | 사용자가 실행하기 쉬움 | 시작 시간·크기·WPF 배포 검증 필요 |

권장안은 Windows x64 자체 포함 폴더 배포다. 별도 설치 프로그램 없이 압축을 풀고 실행할 수 있도록 한다. 이후 실제 용량과 시작 속도를 측정해 단일 파일 여부를 결정한다.

### P2-3. 접근성과 사용성 점검 필요

- 메뉴의 키보드 탐색과 포커스 순서
- 고배율 디스플레이와 Windows 배율 125/150/200%
- 긴 파일명과 작은 창 크기
- 화면 읽기 프로그램용 이름
- 버튼의 선택 상태 및 툴팁 일관성

## 7. 배포 승인 체크리스트

아래 항목이 모두 충족되기 전에는 정식 배포로 판정하지 않는다.

- [ ] DOC 지원을 표시한다면 실제 DOC 왕복 테스트가 통과했다.

## 8. 최종 권고

지금은 기능 확장보다 안정화가 우선이다. 다음 개발자는 먼저 중첩 인라인과 붙여넣기 블록을 처리하고, 그 동작을 자동 검증해야 한다. 이후 실제 Microsoft Word와 LibreOffice에서 DOCX 호환성을 확인한 다음 DOC 지원 및 배포 방식을 확정한다.

## 9. 현재 시점의 다음 실행 계획

V100-P01~P03과 V100-P06~P09가 구현됐다. `BUG-V100-001` 수정도 현재 작업 폴더에 반영되어 자동 회귀 테스트 38건이 모두 통과한다. 다음 단계는 추가 개발이 아니라 테스터의 독립 재검증과 수동 UI 검증이다.

### 개발자 작업 — V100-P11 구현 확인

패치 ID: `V100-P11`  
관련 결함: `BUG-V100-001`  
우선순위: P0  
현재 상태: 구현 및 개발 환경 자동 테스트 통과, 테스터 승인 대기  
목표: 비지원 편집 블록으로 저장이 차단될 때 사용자가 원인과 해결 방법을 알 수 있게 한다.

1. `UnsupportedEditorContentException`을 미지원 콘텐츠 오류로 분류한다.
2. Table, List, Section 등 감지된 블록 종류를 사용자 안내에 포함한다.
3. “일반 텍스트로 다시 붙여넣기”처럼 사용자가 취할 수 있는 해결 방법을 표시한다.
4. 저장은 계속 중단하며 기존 원본과 임시 파일을 변경하지 않는다.
5. 현재 실패 중인 테스트를 통과시키고 전체 회귀 테스트를 실행한다.

완료 조건:

- 자동 회귀 테스트 38건 이상 전부 통과
- 비지원 블록 종류와 해결 방법이 사용자 메시지에 표시됨
- 기존 원본이 변경되지 않음
- 새로운 외부 패키지를 추가하지 않음

개발 완료 산출물:

- 수정된 프로그램 소스
- 통과한 자동 회귀 테스트
- `docs/project/버전_이력.md` 패치 상태 갱신
- `docs/test-reports/테스트_결과_v1.0.0.md` 결과 추가

### 지금 진행할 작업 — 테스터의 V100-P11 재검증 및 수동 UI 검증

1. 개발자가 수정한 빌드에서 전체 자동 테스트를 독립 재실행한다.
2. Table, List, Section 각각의 저장 차단 메시지를 확인한다.
3. 새 문서, 열기, 저장, 다른 이름으로 저장, 저장/폐기/취소 흐름을 직접 조작한다.
4. 굵게, 기울임, 밑줄, 글자 크기, 문단 정렬의 단일·혼합 상태를 확인한다.
5. 결함이 남으면 배포 보류 상태를 유지하고 개발자에게 돌려보낸다.

### 배포·호환성 검증 3차 작업 — V100-P04~P05

Microsoft Word와 LibreOffice를 사용할 수 있는 테스트 환경에서 진행한다.

1. 생성 DOCX를 Word와 LibreOffice에서 복구 경고 없이 여는지 확인한다.
2. 두 프로그램에서 다시 저장한 DOCX를 편집기가 읽는지 확인한다.
3. Word 설치 환경에서 DOC 열기·수정·저장·재열기를 검증한다.
4. 변환 후 `WINWORD.EXE`와 임시 파일이 남지 않는지 확인한다.
5. 실패 항목은 재현 절차와 증거를 포함해 개발자에게 반환한다.

### 최종 배포 단계 — V100-P10

V100-P10은 지금 진행하지 않는다. 에디터 기능, 자동 테스트, 수동 UI 테스트, Word/LibreOffice 호환성 및 DOC 왕복 검증이 모두 끝난 뒤 배포 직전에만 시작한다.

1. 모든 P0/P1 패치와 테스트 결과를 확인한다.
2. Windows x64 자체 포함 Release 결과물을 만든다.
3. 그 결과물을 포함하는 일반 EXE 설치 프로그램을 제작한다.
4. 개발 도구와 .NET이 없는 깨끗한 Windows 10/11 PC에서 설치·실행·제거한다.
5. 설치 후 사용자는 시작 메뉴나 바탕 화면의 `SimpleWordEditor.exe`를 실행하게 한다.
6. 설치 EXE와 SHA-256 파일은 Git 저장소가 아니라 GitHub Release에 첨부한다.
7. 배포 승인 전에는 v1.0.0 태그나 정식 Release를 만들지 않는다.

개발 중에는 Debug/Release 빌드 결과를 직접 실행한다. 설치와 제거를 반복하지 않는다.

### 다음 기능 버전 제한

v1.0.0이 배포 승인되기 전에는 v1.1.0 기능 개발을 시작하지 않는다. 표, 이미지, PDF, 인쇄, 찾기/바꾸기 등은 v1.0.0 범위에 추가하지 않는다.
