# 간단 워드 편집기 버전 이력

이 문서는 버전별 개발 목표, 실제 변경 사항, 수정 패치 및 검증 결과를 누적 관리한다. 아직 배포되지 않은 내용은 “예정”, 구현됐지만 배포되지 않은 내용은 “개발 완료”, 사용자에게 제공된 버전만 “배포 완료”로 표시한다.

## 버전 관리 규칙

버전은 `주버전.부버전.패치버전` 형식으로 관리한다.

- 주버전: 호환성이 크게 달라지거나 제품 구조가 바뀔 때 증가한다. 예: `v1.x.x` → `v2.0.0`
- 부버전: 기존 기능을 유지하면서 새 기능을 추가할 때 증가한다. 예: `v1.0.3` → `v1.1.0`
- 패치버전: 기능 추가 없이 결함, 데이터 안정성, 보안, 호환성 또는 문구를 수정할 때 증가한다. 예: `v1.0.0` → `v1.0.1`

버전이 올라가면 뒤쪽 숫자는 0으로 초기화한다.

## 문서 작성 규칙

앞으로 개발 기획서의 파일명은 다음 형식으로 작성한다.

- 정식/기능 버전 기획서: `개발_기획서_v1.1.0.md`
- 패치 버전 기획서: `패치_기획서_v1.0.1.md`
- 버전별 테스트 보고서: `테스트_결과_v1.0.1.md`
- 전체 누적 이력: `버전_이력.md`

각 버전 기획서에는 반드시 다음 항목을 둔다.

1. 버전 번호와 상태
2. 이전 버전과 달라지는 점
3. 추가되는 기능
4. 수정되는 결함
5. 제외되는 범위
6. 파일 또는 구성 요소별 변경 대상
7. 테스트 항목
8. 완료 및 배포 승인 조건
9. 알려진 제한 사항

개발이 완료되면 “예정 변경”을 실제 구현 결과에 맞게 수정한다. 계획했지만 빠진 항목은 삭제하지 않고 “다음 버전으로 연기”라고 기록한다.

### 배포 방식 결정

- 개발 중에는 설치 프로그램을 만들지 않는다.
- 개발자는 프로젝트 빌드 결과를 직접 실행해 디버깅한다.
- v1.0.0 기능과 품질 검증이 완료된 뒤 V100-P10에서 설치 프로그램을 제작한다.
- 설치 프로그램은 자체 포함 Windows x64 앱을 설치해 사용자가 별도 .NET 설치 없이 실행할 수 있게 한다.
- 설치 EXE는 Git에 커밋하지 않고 GitHub Release 자산으로 배포한다.
- 별도 배포 서버는 운영하지 않는다.
- ZIP 휴대용 배포본은 기본 사용자 배포 방식으로 사용하지 않는다.

### 배포 승인 조건

- P0 패치가 모두 완료되고 회귀 테스트가 통과해야 한다.
- Word와 LibreOffice에서 DOCX가 복구 경고 없이 열려야 한다.
- DOC 지원을 표시하려면 Word 설치 환경의 DOC 왕복 시험이 통과해야 한다.
- 화면에 보이던 본문이 경고 없이 저장 결과에서 사라지는 경로가 없어야 한다.
- 개발 도구가 없는 깨끗한 Windows PC에서 배포본이 실행되어야 한다.
