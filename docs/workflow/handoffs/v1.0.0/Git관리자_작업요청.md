# v1.0.0 Git 관리자 작업 요청

상태: 소스 관리 진행 가능 / 정식 배포 작업 대기  
배포 상태: 보류

## Git 관리자에게 전달할 메시지

현재는 개발 단계입니다. 소스, 테스트, 기획서와 테스트 보고서는 Git으로 관리하되 EXE, DLL, ZIP, 설치 프로그램은 커밋하지 마세요.

`BUG-V100-001`이 해결되고 모든 검증이 끝나기 전에는 v1.0.0 정식 태그나 GitHub Release를 만들면 안 됩니다.

## 지금 해야 할 일

1. 변경 파일이 역할별 산출물 위치에 맞는지 확인합니다.
2. `bin/`, `obj/`, `.localappdata/`, EXE, DLL, ZIP, 설치 결과가 추적되지 않는지 확인합니다.
3. V100-P11 구현과 테스트 문서가 같은 변경 단위를 가리키는지 확인합니다.
4. 테스트가 실패한 상태를 성공으로 표시한 문서가 없는지 확인합니다.
5. 커밋 메시지에 버전과 패치 ID를 포함합니다.

권장 커밋 메시지 예:

```text
fix(v1.0.0): V100-P11 비지원 블록 오류 안내 수정
test(v1.0.0): V100-P11 회귀 및 UI 검증 결과 기록
docs(v1.0.0): 배포 정책과 역할별 인수인계 정리
```

## 지금 하면 안 되는 일

- 설치 EXE 제작 또는 커밋
- 배포 ZIP 커밋
- `v1.0.0` 정식 태그 생성
- GitHub 정식 Release 게시
- 실패 테스트가 남은 커밋을 배포 후보로 표시

## 최종 배포 승인 후 할 일

1. 기획자의 배포 승인 기록을 확인합니다.
2. 승인된 커밋에 `v1.0.0` 태그를 생성합니다.
3. 같은 태그에서 Windows x64 자체 포함 빌드와 설치 EXE를 제작합니다.
4. 테스터가 설치·실행·제거 및 깨끗한 PC 실행을 확인하게 합니다.
5. 설치 EXE와 SHA-256 파일을 GitHub Release 자산으로 첨부합니다.
6. GitHub 자동 소스 ZIP과 설치 파일의 차이를 릴리스 설명에 적습니다.
7. 설치 파일은 Git 저장소 이력에 추가하지 않습니다.
