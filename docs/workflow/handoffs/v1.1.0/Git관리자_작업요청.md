# v1.1.0 Git 관리자 작업 요청

상태: 개발 전 기준점 확인 진행 가능  
정식 배포: 금지

## 개발 시작 전에 할 일

1. v1.0.0의 최신 테스트 통과 상태와 문서 변경이 Git에 반영됐는지 확인합니다.
2. 작업 트리에 다른 역할의 미완료 변경이 남아 있으면 소유자와 내용을 확인합니다.
3. 소스, 테스트, 기획서, 테스트 보고서가 같은 기준 상태를 가리키는지 확인합니다.
4. v1.1.0 작업 브랜치가 필요하면 `codex/v1.1.0-find`처럼 기능 범위가 드러나는 이름을 사용합니다.
5. 현재 기준점을 보존한 뒤 개발자에게 V110-F01 착수 가능 상태를 알려줍니다.

## 개발 중 할 일

- V110-F01과 무관한 변경이 섞이지 않았는지 확인합니다.
- EXE, DLL, ZIP, 설치 파일과 빌드 산출물을 커밋하지 않습니다.
- 개발자의 자체 테스트 결과만으로 정식 완료 태그를 만들지 않습니다.
- 테스터 검증 전 커밋은 개발 완료가 아니라 검토 대상으로 관리합니다.

## F01 테스트 통과 후 할 일

1. 테스터 결과가 `docs/test-reports/테스트_결과_v1.1.0.md`에 있는지 확인합니다.
2. 결함이 없거나 승인된 상태인지 확인합니다.
3. 기능 코드, 테스트, 문서를 함께 추적 가능한 커밋으로 정리합니다.
4. 다음 기능 F02를 시작할 수 있는 깨끗한 기준점을 만듭니다.

권장 커밋 메시지:

```text
feat(v1.1.0): V110-F01 문서 찾기 추가
test(v1.1.0): V110-F01 검색 회귀 결과 기록
```

## 금지 사항

- v1.1.0 정식 태그 생성
- GitHub Release 생성
- 설치 EXE 또는 ZIP 커밋
- Word 호환성 미검증 상태를 배포 승인으로 표시

## 정식 태그와 Release 시점

F01~F04 개발과 독립 테스트가 끝난 뒤에도 바로 Release하지 않는다. 보류된 Word/LibreOffice/DOC 검증, 설치 프로그램 검증 및 기획자 배포 승인이 모두 완료된 후에만 Git 관리자가 태그와 GitHub Release를 만든다.
