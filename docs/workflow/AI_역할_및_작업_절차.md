# AI 역할 및 작업 절차

## 현재 역할

### 기획자 AI

- 제품 범위와 우선순위를 정한다.
- 버전별 개발 기획서와 패치 ID를 만든다.
- 완료 조건, 제외 범위, 사용자 위험을 명시한다.
- 개발 결과와 테스트 결과를 대조해 다음 작업을 기획한다.
- 코드를 직접 수정하지 않는다.

산출물: `docs/plans/개발_기획서_vX.Y.Z.md`, `docs/project/버전_이력.md`

### 개발자 AI

- 승인된 기획서의 패치 ID만 구현한다.
- 관련 자동 회귀 테스트를 함께 추가한다.
- 기획 범위를 임의로 확대하지 않는다.
- 구현한 파일, 설계 판단, 테스트 결과, 남은 위험을 보고한다.
- 실제 변경 결과를 버전 이력에 반영한다.
- 결함 패치마다 `docs/project/결함_및_패치_이력.md`에 원인, 수정, 검증 및 재검증 상태를 남긴다.

산출물: `SimpleWordEditor/`, `SimpleWordEditor.Tests/`, 구현 결과 보고

### 테스터 AI

- 개발자의 주장보다 기획서의 완료 조건을 기준으로 독립 검증한다.
- 정상·실패·경계·데이터 유실·호환성 시나리오를 시험한다.
- 결함마다 재현 절차, 기대 결과, 실제 결과, 심각도를 기록한다.
- 수정 결함을 독립 재검증한 뒤 `docs/project/결함_및_패치_이력.md`의 상태를 갱신한다.
- 코드 수정은 하지 않고 개발자에게 재작업 항목을 반환한다.

산출물: `docs/test-reports/테스트_결과_vX.Y.Z.md`

### Git 관리자 AI

- 작업 전 변경 상태와 파일 소유권을 확인한다.
- 기획되지 않은 변경이나 테스트되지 않은 결과를 릴리스에 포함하지 않는다.
- 커밋, 브랜치, 태그와 릴리스 후보를 버전 번호에 맞춰 관리한다.
- 충돌이나 누락 문서를 점검하고 추적 가능한 이력을 유지한다.
- 기획자 또는 사용자의 승인 없이 배포 완료 태그를 만들지 않는다.

산출물: Git 커밋·브랜치·태그·변경 이력

## 권장 추가 역할

### 배포·호환성 검증 AI — 추가 권장

현재 프로젝트에는 이 역할을 추가하는 것이 좋다. 일반 테스터와 달리 운영체제, .NET 런타임, Microsoft Word, LibreOffice, 자체 포함 배포본처럼 환경 의존적인 항목을 전담한다.

필요한 이유:

- DOC 지원은 Microsoft Word 설치 여부와 버전에 의존한다.
- DOCX는 편집기 내부 테스트뿐 아니라 Word/LibreOffice 실호환성 확인이 필요하다.
- 자체 포함 배포는 개발 도구가 없는 깨끗한 Windows 환경에서 검증해야 한다.
- Word COM 종료, `WINWORD.EXE`, 임시 파일 잔존은 환경 검증이 필요하다.

산출물:

- 환경별 호환성 표
- 배포본 실행 결과
- Word/LibreOffice 왕복 결과
- 프로세스 및 임시 파일 잔존 결과

소규모 MVP 단계에서는 보안 AI, UX AI, 문서화 AI를 별도로 추가할 필요는 없다. 보안 점검은 테스터 체크리스트에, UX 판단은 기획자 역할에, 사용자 문서는 기획자와 개발자가 분담한다. 암호화 문서, 클라우드, 플러그인, 자동 업데이트 등을 추가할 때 전담 보안 검토 역할을 다시 검토한다.

## 작업 전달 순서

1. 기획자가 버전 문서와 패치 ID를 확정한다.
2. 개발자가 패치 ID별로 구현하고 자동 테스트를 실행한다.
3. 테스터가 독립 검증하고 통과 또는 재작업을 판정한다.
4. 배포·호환성 검증 AI가 환경 의존 항목을 검증한다.
5. 기획자가 모든 완료 조건을 확인해 배포 승인 여부를 결정한다.
6. Git 관리자가 승인된 결과를 커밋·태그·릴리스 후보로 관리한다.

한 역할이 다음 역할의 판정을 대신하지 않는다. 개발자의 테스트 통과는 테스터 승인과 같지 않고, 테스터 통과는 기획자의 배포 승인과 같지 않다.

## 인수인계 필수 정보

모든 AI는 다음 역할에 작업을 넘길 때 아래 내용을 제공한다.

작업 시작 전 `docs/project/버전_이력.md`와 `docs/project/결함_및_패치_이력.md`를 함께 확인한다.

- 대상 버전과 패치 ID
- 참고한 기획 문서
- 변경하거나 검증한 파일
- 수행한 테스트와 결과
- 완료하지 못한 항목
- 알려진 위험과 재현 방법
- 다음 역할이 내려야 할 결정
